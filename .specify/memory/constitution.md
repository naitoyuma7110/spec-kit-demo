# TODO App Constitution

## Core Principles

### I. Library-First

すべての機能は独立したライブラリとして設計・実装する。ライブラリは自己完結・独立テスト・明確な目的を持つこと。

### II. Test-First（非交渉）

テスト駆動開発（TDD）を厳守する必要はないが、実装とセットでテストも作成する。実装コードと対応するテストコードを同時に作成し、実装の動作を検証可能にする。ただし、セットアップ段階（000-setup）ではテストは不要。

### III. Integration Testing

新規ライブラリ・契約変更・サービス間連携・共通スキーマなど、統合テストを重視する。実環境に近い形での検証を優先。

### IV. Observability & Simplicity

すべての入出力はデバッグ可能なテキスト形式とし、構造化ログを推奨。設計は常にシンプルを優先し、不要な抽象化や将来のための拡張は避ける。

## Additional Constraints

- セキュリティ要件：ユーザー認証・認可は必須。パスワード等の機密情報は暗号化保存。
- パフォーマンス：初期実装はシンプルさ優先だが、明確なボトルネックが判明した場合のみ最適化を検討。
- 技術スタック：OSSを優先し、依存ライブラリは最小限とする。

## Development Workflow

- すべてのコード・仕様・計画・タスクはバージョン管理し、PRレビューを必須とする。
- 仕様・計画・タスク・実装の各段階でAIによる一貫性チェックを行う。
- デプロイ前に必ず統合テスト・ユーザーテストを実施。

## Governance

- この憲法はすべての開発活動に優先し、変更には明確な理由・記録・メンテナ承認・後方互換性評価が必要。
- すべてのPR・レビューで本憲法への準拠を確認すること。
- 複雑な設計・例外的な実装は必ず理由を記録し、合意を得ること。
